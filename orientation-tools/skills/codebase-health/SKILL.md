# Codebase Health

You are analyzing a codebase to identify high-risk areas before significant work begins. The output is a factual, data-driven report — not a critique. The goal is to help the developer know where to go carefully.

## Step 1: Setup

Determine the target path:
- If the user passed `--path <dir>`, use that directory. Otherwise, use the current working directory.

Determine the time window:
- If the user passed `--since <period>` (e.g. `3 months ago`, `1 year ago`), use that. Otherwise, default to `6 months ago`.

Check that `scc` is available:
- Run `command -v scc`. If not found, tell the user: "This skill requires `scc` for complexity analysis. Install it with `brew install scc`, then re-run." Stop here.

Ensure the `.orientation/` directory exists in the target path. Create it if it doesn't.

Store:
- `TARGET` — the target directory path
- `WINDOW` — the git `--since` value (e.g. `6 months ago`)
- `BASENAME` — the basename of `TARGET` (e.g. `my-project`)
- `TODAY` — today's date (YYYY-MM-DD)

## Step 2: Data Collection

Run the following three analyses. Where possible, use parallel Bash tool calls.

### Analysis A — Churn (files changed most frequently)

```bash
git -C [TARGET] log --since='[WINDOW]' --format='' --name-only \
  | grep -v '^$' \
  | sort | uniq -c | sort -rn \
  | head -50
```

This produces: `<count> <filepath>`. Store the top 50 as the churn list.

Also capture the total number of commits in the window for context:
```bash
git -C [TARGET] log --since='[WINDOW]' --oneline | wc -l
```

### Analysis B — Complexity (cyclomatic complexity per file)

```bash
scc --format json [TARGET]
```

Parse the JSON. It is an array of language objects. Each language has a `Files` array. Each file entry has:
- `Location` — file path
- `ComplexityCount` — cyclomatic complexity
- `Lines` — total lines

Extract all files with `ComplexityCount > 0`. Sort descending by `ComplexityCount`. Store the top 50 as the complexity list.

Exclude vendored, generated, and test directories (e.g. `vendor/`, `node_modules/`, `.git/`, `dist/`, `build/`, `__pycache__/`, files ending in `_test.go`, `.test.ts`, `spec.`).

### Analysis C — Co-change coupling (files that change together)

For each of the top 10 churn files from Analysis A, find which other files appear most often in the same commits:

```bash
# Get all commit hashes that touched a given file
git -C [TARGET] log --since='[WINDOW]' --format='%H' -- [file] | while read hash; do
  git -C [TARGET] show --name-only --format='' "$hash"
done | grep -v '^$' | grep -v "^[file]$" | sort | uniq -c | sort -rn | head -5
```

Do this for the top 5 churn files. Report the top co-change partners for each. This reveals implicit coupling — files that aren't explicitly imported but always change together.

## Step 3: Quadrant Classification

Classify files that appear in both churn and complexity lists:

1. Compute a normalized rank for each file:
   - **Churn rank**: position in the churn list (1 = most changed). Files not in the top 50 churn list get rank 51.
   - **Complexity rank**: position in the complexity list. Files not in the top 50 get rank 51.

2. Apply thresholds:
   - **High churn**: churn rank ≤ 12 (top 25% of 50)
   - **High complexity**: complexity rank ≤ 12

3. Assign quadrant:
   - **Danger Zone**: high complexity + high churn → highest risk
   - **Watch Closely**: low complexity + high churn → simple now, but complexity may grow
   - **Refactor Candidates**: high complexity + low churn → complex but stable, safe to refactor later
   - **Solid Ground**: low complexity + low churn → safe to depend on

Only include files in the quadrant table if they appear in at least one of the two top-50 lists. Focus on source files, not config files.

## Step 4: Write `.orientation/health-report.md`

Write the file to `[TARGET]/.orientation/health-report.md` using this structure:

```markdown
# Health Report — [BASENAME] — [TODAY]

> Generated by `/codebase-health` analyzing [WINDOW] of git history ([N] commits).

## Summary

[2-3 sentences. What does the overall picture look like? Are there a few concentrated danger files, or is risk spread across the codebase? Is churn high relative to the window, or mostly stable?]

| Quadrant | Files |
|----------|-------|
| 🔴 Danger Zone (high complexity + high churn) | [files, comma-separated] |
| 🟡 Watch Closely (low complexity + high churn) | [files] |
| 🟠 Refactor Candidates (high complexity + low churn) | [files] |
| 🟢 Solid Ground (low complexity + low churn) | [representative stable files] |

## Danger Zone Files

Files that are both complex and frequently changed — plan carefully before touching these.

| File | Complexity | Churn (last [WINDOW]) | Notes |
|------|-----------|----------------------|-------|
| [filepath] | [score] | [count] changes | [one line: what this file does] |

[Read each Danger Zone file briefly to provide the Notes column. Keep it to one line.]

## Co-Change Clusters

Files that consistently appear in the same commits are implicitly coupled — a change to one likely requires a change to the others, even without an explicit import relationship.

[For each of the top churn files that has notable co-change partners, list them:]

**[file A]** often changes with:
- [file B] ([N] shared commits)
- [file C] ([N] shared commits)

[If no meaningful co-change patterns found, note: "No strong co-change coupling detected in this window."]

## Refactor Candidates

High complexity but low churn — these files are complex but stable. Safe to refactor when ready, but not urgent.

| File | Complexity | Last changed |
|------|-----------|-------------|
| [filepath] | [score] | [approximate — from git log --follow] |

## Recommended Approach

[3-5 bullets tailored to what was found. Examples:]
- Before modifying [Danger Zone file], add integration tests — it changes frequently and has high cyclomatic complexity.
- [File A] and [File B] are tightly coupled (co-change cluster) — plan changes to both in the same PR.
- The [module/area] is the most volatile part of the codebase. Prefer smaller, incremental PRs in this area.

```

## Step 5: Update `.orientation/README.md` and present summary

### Update the index

If `.orientation/README.md` exists:
- Read it
- Find the `## Files` table
- Add a row for `health-report.md` if not already present:
  `| health-report.md | Complexity × churn analysis — high-risk areas before a refactor |`
- Rewrite the file

If `.orientation/README.md` does not exist, create it:

```markdown
# Codebase Orientation: [BASENAME] -- [TODAY]
Orientation to the codebase providing a breakdown of the structure, architecture and relationships. See the following files for specifics.

## Files
| File | Description |
|------|-------------|
| health-report.md | Complexity × churn analysis — high-risk areas before a refactor |
```

### Present summary

After writing the files, present a concise summary to the user:

1. **Danger Zone**: list the files, with one-line context on each
2. **Top co-change cluster**: the most notable coupling pair/group
3. **Overall risk signal**: high / moderate / low — your assessment of how risky the planned work area is

Keep it brief. The full detail is in `health-report.md`.

Suggest next steps:
- If the user has a specific task in mind: "Run `/codebase-mikado --goal \"[their goal]\"` to trace which of these files your change will touch."
- If they haven't oriented yet: "Run `/codebase-cartographer` to get a full structural map alongside this health data."
